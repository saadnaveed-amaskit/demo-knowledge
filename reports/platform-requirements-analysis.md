## Summary

Retail Nucleus is a **B2B retail pricing intelligence platform** for The Children's Place / Gymboree brands. It lets a **Pricing Team** define reusable product scopes ("Focus Sets") and run AI/ML-driven pricing tools against them — discount modeling, price-scenario optimization, promotions scheduling, cold-start (new-product) entry pricing, and matched-cluster A/B measurement. A **Pricing Strategist** governs the financial risk of those actions through Guardrails, an Approvals queue, an AI-agent roster, and a graduated **Autonomy** trust ladder (from human-recommend to fully autonomous execution). An **Admin** manages user access (currently a stub).

The source `PLATFORM-REQUIREMENTS.md` was reverse-engineered from a design-reference prototype. Its defining characteristic for planning: **the core "intelligence" is not real anywhere yet.** Discount Modeling, Price Scenarios, and Like-Item Mapping all fake their ML/forecasting with deterministic formulas. The documented output *shapes* (types) are a credible target contract for what a real backend must produce, but the modeling itself is entirely unbuilt. Guardrails are advisory (never block), several CRUD operations are wired-in-UI-only, and the closed loops between Measurement → Autonomy → Approvals are narrated but not implemented.

## Personas

The platform names three primary personas in `§1`. The constitution additionally recognizes **Executive** and **Marketing** as platform-wide personas, and several requirements imply a **Merchant** role, so both are flagged below.

### Pricing Team
- **Primary goals:** Build product scopes and run pricing experiments/models to propose price and promotion changes that hit revenue/margin/sell-through targets.
- **Key workflows:** Focus Builder (create/manage Focus Sets) → Product Grid (curate SKUs) → Discount Modeling / Price Scenarios (simulate) → submit for approval; Promotions scheduling; Like-Item Mapping validation; Measurement experiment setup.
- **Decision rights:** Creates and submits scenarios/discount models; does **not** approve them. Can override guardrails only where a guardrail is marked overridable.
- **Unclear assumptions:** Whether "Pricing Team" and "Merchant" (the persona validating Like-Item Mappings, §13) are the same role. `[NEEDS CLARIFICATION]`

### Pricing Strategist
- **Primary goals:** Govern financial risk. Author constraints, approve/deny submissions, and control how much autonomy AI agents have.
- **Key workflows:** Guardrails authoring; Approvals review queue (scenarios + discounts); Agents roster management (hire/pause/retire); Autonomy trust-ladder promotion/demotion, live-action supervision, emergency kill switch.
- **Decision rights:** Sole approver of scenarios and discount models. Sole authority over guardrail definitions and agent autonomy levels. Holds the kill switch.
- **Unclear assumptions:** Whether the Strategist also sets up Measurement experiments (§12 refers to "the strategist" reviewing AI-proposed arms) or whether that is a Pricing Team task. `[NEEDS CLARIFICATION]`

### Admin
- **Primary goals:** Manage user access and brand/division scope.
- **Key workflows:** Configuration screen (currently read-only list of users).
- **Decision rights:** Would own user provisioning/role assignment in production; today none exist.
- **Unclear assumptions:** Whether brand/division access is a real security requirement enforced platform-wide, or display-only. `§14`/`§15.6` flag this as unresolved. `[NEEDS CLARIFICATION]`

### Executive (constitution persona; implied by requirements)
- **Primary goals:** See measurement outcomes and cross-domain impact at a summary level.
- **Key workflows:** §12 notes an executive-level summary of measurement outcomes is intended to live in the **Performance Platform**, not built anywhere yet.
- **Unclear assumptions:** Whether any Executive-facing surface is in scope for *this* platform's rebuild. `[NEEDS CLARIFICATION]`

### Marketing (constitution persona; implied by requirements)
- **Primary goals:** Own campaigns/discount tiles referenced in Deep Dive and cross-domain actions.
- **Key workflows:** Deep Dive Marketing/Discount tiles; cross-domain sub-actions surfaced into Approvals; the whole Marketing module in DATA-TAXONOMY.
- **Unclear assumptions:** The Marketing module is heavily represented in DATA-TAXONOMY but **absent from PLATFORM-REQUIREMENTS** (which scopes to `/retail-nucleus/*` only). Is Marketing in scope for this rebuild? `[NEEDS CLARIFICATION]`

## Product Domains

### 1. Scoping (Focus Sets) — foundational
- **Description:** Named, reusable product/SKU scope defined by a nestable AND/OR condition tree. Every downstream pricing tool targets a Focus Set.
- **Personas:** Pricing Team.
- **Frontend surfaces:** Focus Builder (`focus`), Product Grid (`product-grid`).
- **Backend capabilities:** Focus Set CRUD + **"resolve to SKUs"** as a first-class operation; catalog attribute enumeration; CSV export.
- **Data entities:** `FocusSetEntity`, `CatalogProductEntity`, `CatalogSkuEntity`, `FocusDiscountEntity`.

### 2. Promotions
- **Description:** Calendar/list scheduling of promotional events tied to a Focus Set and discount terms.
- **Personas:** Pricing Team.
- **Frontend surfaces:** Promotions (`promos`) — calendar + list + detail drawer.
- **Backend capabilities:** Promo CRUD; derive promo status from dates; compute per-SKU promo price.
- **Data entities:** `PromoEntity` (+ the fragmented `Promo`/`FocusDiscountEntity`/`DiscountModelEntity` reconciliation, §15.1).

### 3. Discount Modeling
- **Description:** AI/ML markdown simulation → full margin/revenue/risk forecast report → submit for approval.
- **Personas:** Pricing Team (build/submit), Pricing Strategist (approve).
- **Frontend surfaces:** Discount Modeling (`discount-modeling`) — list, create form, run sequence, output view with charts + rollup + 6 risk panels.
- **Backend capabilities:** Run simulation producing `DiscountModelOutput`; persist model + status lifecycle.
- **Data entities:** `DiscountModelEntity`, `DiscountModelOutput` and sub-shapes.

### 4. Price Scenario Optimization
- **Description:** ML what-if pricing against a Focus Set; profit-vs-revenue frontier with an optimization-level slider; SKU-level Deep Dive.
- **Personas:** Pricing Team (build/submit), Pricing Strategist (approve).
- **Frontend surfaces:** Price Scenarios (`scenario`), Deep Dive (inline).
- **Backend capabilities:** Run ML pipeline (6 phases) producing frontier + uplift + narrative + per-SKU recommendations; live guardrail evaluation.
- **Data entities:** `ScenarioEntity`, `DeepDiveView`, `DeepDiveProductView`, `DeepDiveMarketingTile`, `DeepDiveDiscountTile`.

### 5. Governance — Guardrails
- **Description:** Strategist-authored margin/markdown/cadence constraints segmented by brand × division.
- **Personas:** Pricing Strategist.
- **Frontend surfaces:** Guardrails (`guardrails`) — table, inline edit, toggles, CRUD.
- **Backend capabilities:** Guardrail CRUD + persistence + **live compliance evaluation** against projected impact; versioning/audit.
- **Data entities:** `GuardrailEntity`.

### 6. Governance — Approvals
- **Description:** Strategist review queue gating scenarios, discount models, and cross-domain sub-actions.
- **Personas:** Pricing Strategist.
- **Frontend surfaces:** Approvals (`approvals`) — two tabs, pending queue, decided table, review drawers.
- **Backend capabilities:** Approval state machine; mandatory-reason enforcement; live guardrail-compliance record; cross-domain action linkage.
- **Data entities:** `ApprovalEntity`, `GuardrailCheck`, `SubActionEntity`, `BusinessIssueEntity`.

### 7. Governance — Agents & Autonomy
- **Description:** Manage the AI pricing "workforce" (Monitors/Operators/Task agents) and govern how much unsupervised authority each *action class* has earned via a trust ladder, with live supervision and an emergency kill switch.
- **Personas:** Pricing Strategist.
- **Frontend surfaces:** Agents (`agents`), Autonomy (`autonomy`).
- **Backend capabilities:** Agent roster CRUD/lifecycle; trust-ladder promotion gates; reversibility ceilings; autonomy caps; live action state machine (Staged → Auto-Pending-Veto → Applied → Vetoed/Rolled Back); audit trail; global kill switch.
- **Data entities:** agent/operator/action-class records (not yet in DATA-TAXONOMY — see Data Taxonomy Notes).

### 8. Measurement (Experimentation)
- **Description:** Matched-cluster A/B testing of price changes with sequential (early-stopping) readout, since shopper-level randomization is impossible.
- **Personas:** Pricing Strategist (and/or Pricing Team).
- **Frontend surfaces:** Measurement (`measurement`) — setup (arm assignment, balance checks, go-live gate) + live (probability trend, verdict banner).
- **Backend capabilities:** Cluster/block generation, balance computation, sequential test estimator, causal-result feedback to Autonomy.
- **Data entities:** `ClusterView`, `BlockView` (Tier-2 today; likely need Tier-1 experiment/outcome entities).

### 9. Cold Start (Like-Item Mapping)
- **Description:** Map new-season products with no history to prior-season comparables to set entry pricing; merchant validates/overrides.
- **Personas:** Merchant / Pricing Team.
- **Frontend surfaces:** Like-Item Mapping (`like-items`) — queue, detail with candidate ranking, confidence breakdown, override.
- **Backend capabilities:** Look-alike/similarity model; PIM/DAM + historical-sales integration; confidence scoring; audit trail.
- **Data entities:** `ColdStartMappingEntity`, `CompCandidateView`, `ImpactPreviewView`.

### 10. Administration
- **Description:** User access + brand/division scoping.
- **Personas:** Admin.
- **Frontend surfaces:** Config (`config`).
- **Backend capabilities:** Identity provider integration (SSO/IAM); user CRUD; brand/division access enforcement.
- **Data entities:** `AppUserEntity`.

## Technology and Design Implications

Constraints future `plan.md` work must obey (not implementation decisions to make here):

- **UI components:** shadcn/ui only. Every Button/Dialog/Select/Table/Tabs/Drawer must use approved shadcn primitives; no hand-rolled equivalents. Detail drawers (Promotions, Approvals, Like-Item) and confirmation dialogs are pervasive.
- **Charts:** Nivo only. Applies to forecast charts (cumulative revenue w/ CI band, gross-margin waterfall, inventory burndown), profit-vs-revenue frontier, probability-of-winning trend, balance visualizations. Inline SVG only for decorative visuals with no Nivo equivalent — the frontier curve and waterfall need verification that Nivo covers them.
- **Large tables:** AG Grid for dense grids (Product Grid, Deep Dive price-adjustments, SKU-level detail, rollup). `.ag-root-wrapper { height: 100% }` required. Smaller tables may use shadcn Table.
- **State:** Zustand (`state.ts`, `useXxxState`) for UI state; **nuqs** for URL-synced filters (Focus Set library search/sort, Product Grid search, Like-Item filters, Approvals tab). Global filters `brand`/`channel` live in `useRNState`.
- **Routing:** React Router v7; routes already enumerated in §1.
- **Forms + validation:** React Hook Form + Zod for all create/edit forms (Focus Set, Promo, Discount Model, Scenario, Guardrail). Date validation (end > start, no past start) is a Zod concern.
- **Animation:** Framer Motion page transitions (`opacity/y` standard, 0.18s easeOut). Run sequences (Discount Modeling / Price Scenario multi-phase progress) are animation surfaces.
- **Dates:** date-fns only — heavily used (promo status derivation, calendar grid, durations, measurement day-by-day).
- **Design system:** Zinc + teal (`#14b8a6`); semantic green/amber/red/blue; Geist Sans/Mono with `RN_TYPOGRAPHY` tokens (no raw `text-*`); Lucide 1.5px icons. Qty/sell-through/risk color thresholds recur — reconcile into canonical scales (§2/§4).
- **Testing:** Vitest + React Testing Library (unit); Cucumber/Gherkin + Playwright (acceptance) where a runnable BDD setup exists. Accessible-role queries preferred.
- **Build/check:** `npm run check` (lint + policy lint + build) must pass where available; `npm run lint:policy` enforces the pinned stack. Any deviation needs a `// tech-policy-exception:` annotation.

## Recommended First Spec

**`focus-set-management` (CF-1, Focus Builder).**

- **Why it is a good first feature:** It is the foundational scoping primitive — Discount Modeling, Promotions, Price Scenarios, and Product Grid all target a Focus Set (§1/§15.2). Nothing downstream is meaningfully testable without it. Its requirements are the most complete and highest-confidence in the document, with clean, well-numbered FRs (FR-FB-01…10) and few open questions relative to the ML-heavy screens.
- **What it validates about the harness:** Exercises the full governed pipeline end-to-end on a tractable feature — spec → plan → tasks → contract-first backend (Focus Set CRUD + resolve-to-SKUs) → acceptance-test-first frontend → validation report — proving the multi-repo gates, the pinned stack, and the data-taxonomy discipline (`FocusSetEntity` Tier-1 vs. `FocusSetView`/`ProductView` Tier-2, which are already defined) before tackling harder domains.
- **Expected target repos:** both. Backend owns the Focus Set resource + resolve-to-SKUs operation + catalog attribute enumeration; frontend owns the builder UI, library, preview, and export.
- **Expected acceptance tests:** create/edit/duplicate/delete a Focus Set; nestable AND/OR condition tree; live preview with match count + "no matches" warning; sort (name/date/count) + case-insensitive search; CSV export; drill to Product Grid.
- **Expected contract needs:** Focus Set CRUD; **resolve Focus Set → SKUs**; catalog attribute enumeration; matched-SKU export payload. Resolve first the §2/§15.1/§15.2 open questions (minimum-match enforcement, live vs. snapshot counts, canonical discount model) during spec authoring.
